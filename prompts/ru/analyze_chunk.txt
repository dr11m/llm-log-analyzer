## Контекст
- Временной диапазон: {chunk_start_time} - {chunk_end_time}
- Количество строк: {chunk_line_count}
- Позиция: Чанк {chunk_position} (1 = самый новый)

## Логи
```
{log_lines}
```

## Задача
Изучи логи и найди всё, что заслуживает внимания:
- Аномальное поведение (отход от нормы, описанной в документации проекта)
- Ошибки, исключения, failures и их частоту
- Деградацию производительности
- Подозрительные паттерны (повторяющиеся ошибки, рост задержек, потеря данных)
- Всё, что по твоему мнению может указывать на проблему

Для каждой найденной проблемы определи:
- Критичность (CRITICAL / MEDIUM / LOW) и обоснуй почему
- Конкретные строки логов как доказательство (с timestamp)
- Как это влияет на работу системы
- Что рекомендуется сделать

## Формат ответа (строго JSON, без markdown-обертки)

{
  "issues": [
    {
      "severity": "CRITICAL",
      "title": "Краткое название проблемы",
      "description": "Подробное описание с конкретными числами. Например: 42 из 50 запросов (84%) вернули ошибку 429 за период 01:08-01:12",
      "evidence": [
        "ПЕРВАЯ строка лога с проблемой (полностью с timestamp)",
        "ПОСЛЕДНЯЯ строка лога с проблемой (полностью с timestamp)"
      ],
      "count": 42,
      "impact": "Как это влияет на работу системы",
      "recommendation": "Что делать"
    }
  ],
  "summary": "2-3 предложения: что происходило в этом временном отрезке, общее состояние",
  "time_range": {
    "start": "первый timestamp из логов",
    "end": "последний timestamp из логов"
  }
}

## Правила
- Одна корневая причина = одна проблема. Не дублируй.
- Указывай ТОЧНЫЕ числа из логов, не "множество" или "массовые".
- evidence: первая и последняя строка проблемы, чтобы видеть временной диапазон.
- Если серьёзных проблем нет — верни пустой issues и напиши в summary, что всё в норме.
- НЕ выдумывай, только то что видно в логах.
